# Custom GitHub Copilot Instructions Templates for Full Repository Access

Based on the current GitHub Copilot architecture, **custom instructions files do not control or grant permissions** to Copilot within repositories. Permission management works differently than you might expect.[^1][^2][^3][^4]

## Understanding Copilot Permissions

GitHub Copilot's access permissions are controlled at the **organization, repository, and user authentication levels**, not through instruction files. The key points:[^5][^3][^4]

**Authentication-Based Access**

- Copilot inherits access based on your GitHub account permissions[^3][^6]
- If you have write access to a repository, Copilot coding agent can work in that repository[^4][^7]
- Private repositories require appropriate organizational settings[^6]

**Organizational Controls**

- Organization owners control which repositories Copilot can access[^8][^4]
- Copilot coding agent must be enabled at the organization level for Business/Enterprise plans[^5][^4]
- Repository-level settings determine if coding agent can work in specific repos[^4]

**Copilot Coding Agent Permissions**
Once enabled, the coding agent has extensive capabilities including:[^7][^9]

- Exploring your codebase
- Making changes across multiple files
- Running builds and tests
- Creating pull requests
- Executing linters and automated checks


## Custom Instructions Templates

While instructions don't grant permissions, they **guide Copilot's behavior**. Here are comprehensive templates:

```
## Coding Standards

### General Guidelines
- Use semicolons at the end of statements
- Use single quotes for strings
- Maximum line length: 100 characters
- Always include JSDoc comments for public functions
- Use async/await instead of callbacks

### Naming Conventions
- **Variables:** camelCase (e.g., `userName`, `isActive`)
- **Constants:** UPPER_SNAKE_CASE (e.g., `MAX_RETRIES`)
- **Functions:** camelCase with verb prefix (e.g., `getUserData`, `handleClick`)
- **Components:** PascalCase (e.g., `UserProfile`, `DataTable`)
- **Files:** kebab-case (e.g., `user-profile.js`, `data-service.js`)

### Code Organization
- Keep functions under 50 lines
- Extract reusable logic into utility functions
- Place helper functions in `/src/utils/`
- Group related functionality into modules

## Branch Protection and Workflow
- **Default Branch:** `main` (protected)
- **Development Branch:** `develop`
- **Feature Branches:** `feature/description`
- **Bug Fix Branches:** `bugfix/description`
- Always create pull requests from feature branches to `develop`
- Never commit directly to `main` branch
- All pull requests require code review before merging

## Testing Requirements
- All new features must include unit tests
- Maintain minimum 80% code coverage
- Integration tests required for API endpoints
- Run `npm test` before pushing changes

## Error Handling
- Use try/catch blocks for all async operations
- Implement proper error boundaries in React components
- Always log errors with contextual information
- Use custom error classes for specific error types

## Security Guidelines
- Never commit API keys, passwords, or secrets
- Use environment variables for sensitive data
- Sanitize all user inputs
- Implement proper authentication and authorization checks

## CI/CD Pipeline
GitHub Actions workflow runs on every pull request:
1. Lint check
2. Unit tests
3. Integration tests
4. Build validation
5. Security scanning

All checks must pass before merging.

## Dependencies
- Keep dependencies up to date
- Review security advisories regularly
- Document any version pinning requirements
- Use exact versions in package.json for production dependencies

## Documentation Requirements
- Update README.md for new features
- Add inline comments for complex logic
- Maintain API documentation in `/docs/api/`
- Update CHANGELOG.md for all changes

## Validation Steps
Before submitting pull requests:
1. Run full test suite: `npm test`
2. Check code coverage: `npm run coverage`
3. Run linter: `npm run lint`
4. Build project: `npm run build`
5. Test in local environment
6. Review all changes manually

## Important Notes
- Trust these instructions unless they are found to be incomplete or incorrect
- When in doubt, search the codebase for existing patterns
- Follow the established patterns in similar files
- Ask for clarification on ambiguous requirements
```


### Path-Specific Instructions Template

**File Location:** `.github/instructions/frontend.instructions.md`

```markdown
---
applyTo: "src/components/**/*.jsx,src/components/**/*.tsx"
description: "React component development guidelines"
---

# React Component Guidelines

## Component Structure
- Use functional components with hooks
- Follow this structure:
  1. Imports
  2. Type definitions/PropTypes
  3. Component function
  4. Helper functions
  5. Styles
  6. Export

## Hooks Guidelines
- Follow React hooks rules (no conditional hooks)
- Custom hooks must start with "use" prefix
- Always include cleanup in useEffect when needed
- Use useCallback for functions passed to child components
- Use useMemo for expensive calculations

## Props
- Use TypeScript interfaces for props
- Always destructure props
- Provide default values for optional props
- Document all props with JSDoc comments

## Styling
- Use CSS modules for component-specific styles
- Follow BEM naming convention
- Use Tailwind CSS utility classes where appropriate
- Keep inline styles minimal

## State Management
- Use local state when possible
- Lift state up only when necessary
- Use Context API for app-wide state
- Document state shape and updates

## Performance
- Memoize expensive computations with useMemo
- Use React.memo for pure components
- Avoid unnecessary re-renders
- Lazy load components when appropriate
```

## Important Limitations and Clarifications

**What Custom Instructions CANNOT Do:**

- Grant or modify repository permissions[^2][^1]
- Override organizational security policies[^8][^3]
- Access private repositories without proper authentication[^6]
- Bypass branch protection rules[^10]
- Modify permission scopes[^11][^3]

**What They CAN Do:**

- Guide Copilot's coding style and conventions[^12][^1][^2]
- Provide project context and structure[^13][^1]
- Specify build and test procedures[^1][^13]
- Define coding standards[^14][^12][^2]
- Improve code generation quality[^15][^14]

**Enabling Full Copilot Access:**
To ensure Copilot coding agent has maximum authorized access to your repositories:[^4]

1. **Enable at Organization Level** (for Business/Enterprise)[^5][^4]
2. **Enable for Specific Repositories**[^4]
3. **Verify Write Access** - Your GitHub account must have write permissions[^7][^4]
4. **Configure Repository Settings** - Allow coding agent in repository settings[^9][^4]

The custom instructions templates above provide comprehensive guidance to Copilot, maximizing its effectiveness within the permissions your GitHub account and organization settings have already granted.[^2][^1][^7]
